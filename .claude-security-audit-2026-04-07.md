## Critical — 必须修复

### C-01: C2 Beacon Shell 命令直接执行用户输入

**文件**: `core/c2/beacon.py:616-632`

**漏洞代码**:
```python
def _handle_shell(self, command: str) -> str:
    """执行 Shell 命令"""
    if platform.system() == "Windows":
        result = subprocess.run(
            ["cmd.exe", "/c", command],  # ← command 未经任何过滤
            capture_output=True, text=True, timeout=self.config.shell_timeout,
        )
    else:
        result = subprocess.run(
            ["/bin/bash", "-c", command],  # ← 同上
            ...
        )
```

**漏洞原理**: `command` 参数由 C2 Server 下发，通过 `["/bin/bash", "-c", command]` 执行。虽然使用了列表形式避免了 `shell=True`，但 `-c` 后的 `command` 仍然在 bash 内部被解析为完整 shell 语句。如果 C2 通信被劫持(MITM)或 C2 Server 被攻陷，攻击者可执行任意命令。

**风险**: C2 通道被 MITM 或投毒时，无任何命令白名单/沙箱保护。

**修复建议**:
1. 实现命令白名单机制 (仅允许预定义的 recon 命令)
2. 通过 `SafeExecutor` 包装执行
3. 对命令内容进行签名验证 (确保来源为可信 C2 Server)

### C-02: Webshell 生成器 shell=True 命令执行

**文件**: `core/persistence/webshell_manager.py:473`

**漏洞代码**:
```python
result = subprocess.check_output(cmd, shell=True, stderr=subprocess.STDOUT)
```

**漏洞原理**: 生成的 Python Webshell 代码中使用 `shell=True`。虽然注释标注了"预期行为"，但该函数 `python_shell()` 生成的代码在测试环境可能被误用。代码中 `cmd` 直接来自 HTTP 请求参数 `request.values.get('cmd')`，无任何过滤。

**修复建议**:
1. 生成的 Webshell 代码中使用 `shlex.split()` + `shell=False`
2. 添加命令黑名单过滤 (如 `rm -rf`, `mkfs`, `dd`)
3. 在模块 docstring 和注释中更明确地标注安全边界

## High — 优先修复

### H-01: dns_resolver 外部命令 check_output 无输入过滤

**文件**: `core/recon/dns_resolver.py:316,358,391`

**漏洞代码**:
```python
cmd = ["nslookup", "-type=mx", domain]  # domain 来自用户输入
output = subprocess.check_output(cmd, stderr=subprocess.DEVNULL, timeout=self.timeout)
```

**分析**: `domain` 参数经过 `@validate_inputs(target="target")` 装饰器在 handler 层验证，但 `dns_resolver.py` 作为 core 层模块可被其他代码直接调用（绕过 handler 层验证）。`nslookup` 本身对恶意域名有一定抵抗力（列表形式避免了 shell 注入），但 **defense-in-depth 不足**。

**修复建议**: 在 `DNSResolver` 构造函数或 `_query_*` 方法内部增加 domain 格式校验。

---

### H-02: `__import__()` 动态导入 — AI Engine Provider 加载

**文件**: `core/ai_engine.py:77`

**漏洞代码**:
```python
mod = __import__(module_name, fromlist=[class_name])
```

**分析**: `module_name` 来自 `provider_map` 硬编码字典 (`"openai"`, `"anthropic"`)，当前不可被用户直接控制。但如果未来扩展 provider 支持用户自定义，将构成任意模块加载漏洞。

**风险等级**: High (设计风险，当前无直接利用路径)

**修复建议**: 维持白名单字典不变；添加断言 `assert module_name in ALLOWED_MODULES`。

### H-03: pickle.dumps RCE 对象序列化

**文件**: `core/exploit/exploiters/deserialize_exploiter.py:80,259`

**漏洞代码**:
```python
class RCE:
    def __reduce__(self):
        return (os.system, (self.cmd,))

payload = base64.b64encode(pickle.dumps(RCE(command))).decode()
```

**分析**: 这是 **预期的攻击功能** — 生成反序列化 RCE payload 用于渗透测试。但该代码：
1. `os.system` 而非 `subprocess.run` — 无超时保护
2. `command` 参数来自 MCP tool 调用，依赖 handler 层 `@require_critical_auth` 保护
3. 如果 auth 被绕过，可直接生成任意命令的 pickle payload

**修复建议**: 确认 auth 层不可绕过（见 H-04）。

### H-04: Auth Middleware PERMISSIVE 模式绕过

**文件**: `core/security/mcp_auth_middleware.py:97-118`

**漏洞代码**:
```python
if _auth_config["mode"] == AuthMode.DISABLED:
    return cast(Dict[str, Any], await func(*args, **kwargs))

if not api_key_str:
    if _auth_config["mode"] == AuthMode.STRICT:
        return {"success": False, "error": "Authorization required..."}
    else:
        # 宽松模式：允许但记录警告
        return cast(Dict[str, Any], await func(*args, **kwargs))
```

**漏洞原理**: 
1. `AuthMode.DISABLED` 直接跳过所有检查
2. `AuthMode.PERMISSIVE` 模式下无 API Key 也能调用 CRITICAL 级工具
3. `set_auth_mode()` 是全局函数，任何代码路径都可调用
4. conftest.py 中 `disable_auth_for_tests` 将模式设为 DISABLED — 如果测试配置泄露到生产环境...

**修复建议**:
1. CRITICAL 级工具 **永远** 要求 API Key（即使 PERMISSIVE 模式）
2. `set_auth_mode(DISABLED)` 仅在 `AUTOREDTEAM_ENV=test` 时允许
3. 在 `mcp_stdio_server.py` 启动时日志显示当前 auth mode

### H-05: subprocess_runner allow_shell=True 注入风险

**文件**: `shared/subprocess_runner.py:58-82`

**漏洞代码**:
```python
if allow_shell:
    use_shell = True
    logger.warning("subprocess 显式启用 shell 模式...")
# ...
result = subprocess.run(cmd, ..., shell=use_shell)
```

**分析**: `allow_shell=True` 参数使字符串命令以 `shell=True` 执行。当 `cmd` 包含用户可控部分时，构成命令注入。目前调用方未传入 `allow_shell=True`，但接口暴露了风险。

**修复建议**: 废弃 `allow_shell` 参数，强制 `shell=False`。需要 shell 特性的场景用 `SafeExecutor`。

## Medium — 需评估修复

### M-01: 大量 SSL 验证禁用

**文件**: 28 处 (`check_hostname=False`, `verify=False`)

**分布**: `core/recon/` (6处), `core/exploit/` (4处), `core/c2/` (3处), `core/stealth/` (4处), `modules/` (5处), 其他 (6处)

**分析**: 渗透测试工具需要能扫描自签名目标，部分 `verify=False` 是预期行为。但：
1. `core/c2/tunnels/http.py` 中 `verify=False` 意味着 C2 通道不验证 Server 证书 — MITM 风险
2. 建议对 C2 通道强制证书固定 (cert pinning) 或至少允许配置

---

### M-02: MD5 用于非安全上下文

**文件**: `core/c2/beacon.py:161`, `core/orchestrator/contracts.py:104,273`, `core/mcts_planner.py:127` 等 (18处)

**分析**: MD5 用于 Beacon ID 生成、缓存键、指纹去重等非安全用途。虽然非密码学使用，但 `auth_manager.py:432` 使用 `sha256` 哈希 API Key 时 **没有加盐** — 容易被彩虹表攻击。

**修复建议**: `_hash_key()` 中加入随机盐并存储。

---

### M-03: `random` 模块用于安全相关场景

**文件**: `core/c2/tunnels/http.py:409`, `core/feedback/engine.py:429-430` 等

**分析**: `random.choice`/`random.randint` 是伪随机，不适合安全场景。C2 隧道中使用 `random.choice` 选择通信方式，可能被预测。

**修复建议**: C2 相关代码使用 `secrets` 模块。

---

### M-04: 审计日志中参数可能泄露敏感信息

**文件**: `core/security/auth_manager.py:420-422`

```python
"params": log.params,  # 直接记录原始参数
```

**分析**: 虽然 `AuthManager` 定义了 `SENSITIVE_FIELDS` 用于脱敏，但审计日志的 `params` 字段是否真正经过脱敏取决于调用方实现。

**修复建议**: 在 `_log_audit()` 中强制对 params 进行脱敏过滤。

---

### M-05: RateLimitTracker 内存无上限

**文件**: `core/security/auth_manager.py:438-457`

```python
self.rate_limit_tracker[key_id] = [t for t in ... if t > hour_ago]
```

**分析**: 每个 `key_id` 维护一个时间戳列表，如果有大量不同 key_id 请求，会导致内存膨胀。虽然 `MCPSecurityMiddleware` 的 `RateLimitConfig` 有 `max_keys=10000`，但 `AuthManager` 的 tracker 没有此限制。

**修复建议**: 添加 `max_keys` 检查，超限时清理最旧的 key。

---

### M-06: Auth 模式可被全局修改

**文件**: `core/security/mcp_auth_middleware.py:53-56`

```python
def set_auth_mode(mode: AuthMode):
    _auth_config["mode"] = mode
```

**分析**: 无任何保护的全局状态修改。任何导入 `set_auth_mode` 的代码都能降级认证。

**修复建议**: 添加 caller 验证或将 auth mode 设为启动时不可变配置。
